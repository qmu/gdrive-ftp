# `find` command: search Drive for files/folders by name (native `name contains` query)

## Overview

gdrive-ftp can navigate by name (`ls`/`cd`), address by ID (`id:`), but cannot
**search**. Add a `find` command â€” the search counterpart â€” that locates
files/folders by name using Google Drive's native server-side query, printing each
match as a full path so the user (or an agent) can then act on it (e.g. `get`/`rm`
by its `id:`).

```
$ gdrive-ftp find report
/My Drive/Work/Quarterly Report.pdf
/My Drive/Archive/report-2025.txt
```

**Confirmed semantics:**
- **Match:** case-insensitive **substring** (`contains`), via Drive's native
  `name contains '<pattern>'` query â€” Google does the matching server-side, no
  folder-by-folder walking. Drive's `contains` is token/normalization-flavored, so
  results are **re-filtered client-side** with a case-insensitive `strings.Contains`
  to guarantee true substring semantics (mirrors the existing `FindChildren`
  client-side re-filter for exact match).
- **Scope:** the **current drive context**, reusing the existing
  `withDrive(call, driveID)` scoping â€” exactly how the API is meant to be used:
  - My Drive context (`driveID == ""`) â†’ default `corpora=user` (My Drive + items
    shared with you).
  - Inside a Shared Drive (`driveID` set) â†’ `corpora=drive` + that `driveId`.
  - This deliberately avoids `corpora=allDrives`, which Google documents as a
    performance concern ("When possible, use the `user` or `drive` corpora instead").
- **Optional anchor:** `find <pattern> [path-or-id:]` resolves the anchor via
  `resolveDir` to retarget the search at a different drive, and â€” when the anchor
  is a folder â€” narrows results to that folder's **subtree** by filtering on
  ancestors client-side (Drive's `contains` query is not folder-scoped; `'<id>' in
  parents` matches only direct children).

**Usage:** `find <pattern> [dir]` â€” `<pattern>` is the substring; optional `[dir]`
(a path or `id:` token) scopes the search. Defaults to the cwd's drive.

## Implementation Steps

1. **Search primitive (`internal/gdrive/client.go`).** Add `Search(ctx, driveID, pattern)` modeled on `List`: build `q := fmt.Sprintf("name contains '%s' and trashed = false", escapeQ(pattern))`, `Spaces("drive")`, `Fields("nextPageToken, files("+fileFields+")")` (keep `parents` for path building), `SupportsAllDrives(true)`, scope with `withDrive(call, driveID)`, paginate via `.Pages`. Returns `[]*drive.File` behind the boundary (consistent with `List`).

2. **Client-side substring re-filter.** Drive's `contains` is not a strict substring match; re-filter the returned files with case-insensitive `strings.Contains(strings.ToLower(f.Name), strings.ToLower(pattern))` so the result matches the documented semantics. Keep this as a small pure helper for unit testing.

3. **`cmdFind` handler (`internal/shell/commands.go`).** Parse `args`: require `<pattern>` (else `usageErr("find <pattern> [dir]")`). Resolve the optional `[dir]` anchor via `s.resolveDir(arg)` â†’ stack (default `s.cwd`); derive `driveID := currentDriveID(stack)`. Call `s.c.Search(s.ctx, driveID, pattern)`.

4. **Subtree narrowing (when an anchor folder is given).** If the anchor resolved to a non-empty folder stack, keep only matches whose ancestor chain includes `currentID(stack)`. Since `*drive.File.Parents` is only the direct parent(s), walk parents upward (cache `GetByID`/parent lookups) or, to bound cost, document the v1 behavior: an anchor scopes the **drive corpus**, and subtree filtering is best-effort one or two levels â€” choose one approach explicitly and note the trade-off rather than silently doing a deep walk.

5. **Full-path rendering.** For each match, build its absolute path (drive name + ancestor folder names + file name), reusing the `pwd()` stackâ†’path join. Resolve ancestor names from `Parents` via `GetByID` (cache to avoid repeat calls). Emit through the JSON ticket's `emit()` seam: text = one path per line (folders suffixed `/`); JSON = an array of `{path,id,name,isFolder,mimeType,size?,modifiedTime?}`.

6. **Empty state.** No matches is an explicit, self-explanatory outcome: in text mode print nothing to stdout and exit 0 (FTP-like) **or** a `no matches` line â€” pick one and document it; in JSON mode emit `[]`. Do not error.

7. **Registration + completion.** Add the `find` entry to the `commands` map with `usage: "find <pattern> [dir]"`, `help: "search for files/folders by name (substring)"`. Add `find` to `argKind` so the `[dir]` arg completes as `remote`.

8. **Tests + docs (same commit).** `TestArgKind` case for `find`; pure-helper tests for the substring re-filter and any path/ancestor helper. Update `README.md` (Commands table + a short `find` note: substring, current-drive scope, optional anchor, output) and `plugins/gdrive-ftp/skills/gdrive-ftp/SKILL.md` (agent-facing: `find <pattern> [dir]`, that results are full paths / JSON objects with `id`, and that you then act on a result via `id:`).

## Considerations

- **Query belongs at the vendor boundary** (`internal/gdrive/client.go`). The `name contains` query, `escapeQ`, `withDrive` scoping, pagination, and error mapping stay inside `internal/gdrive`; `cmdFind` only parses scope args and formats output. No `srv.Files` calls leak into the shell. (`workaholic:implementation` Domain Layer Separation + Conservative Vendor Dependence.)
- **`find` is non-destructive and must never feed a guessed target into a mutating op** (`internal/shell/commands.go`). Unlike `rm`/`cd`/`put`, which refuse on `ErrAmbiguous`, `find` is a multi-result query: present *all* matches with enough identity (full path + `id`) that the user acts unambiguously afterward via `id:`. `find` must not collapse to one result or trigger destructive behavior. (`workaholic:design` reconciliation of the exact/ambiguous-match safety stance.)
- **API corpus choice is the idiomatic one** (`internal/gdrive/client.go`). Default `corpora=user` and per-`driveId` `corpora=drive` (via `withDrive`) are what Google recommends; `corpora=allDrives` is explicitly avoided per Drive's performance guidance. Searching a Shared Drive requires the cwd/anchor to be in that drive.
- **Output is one contract with `-json`** (`internal/shell/output.go`). `find` routes through the `emit()` seam and the owned DTO from the JSON ticket (hence `depends_on`), adding a `path` field â€” so `gdrive-ftp -json find report` yields a parseable array, and text/JSON expose the same fields. (`workaholic:implementation` Accessibility for Humans and AI.)
- **Self-explanatory command + designed empty state** (`internal/shell/commands.go`, `README`). Register `usage`/`help`, mirror `ls` row/path conventions, and make "no matches" an explicit outcome, not a confusing blank. (`workaholic:design` Self-Explanatory UI.)
- **Path-building cost** (`internal/shell/commands.go`, `internal/gdrive/client.go`). Rendering full paths needs ancestor-name lookups from `Parents` (extra `GetByID` calls); cache per-folder-ID within a single `find` to bound API calls on large result sets. (`workaholic:implementation` Capacity/Recovery â€” keep the cost model simple and explicit.)
- **Testability debt** (`internal/shell/shell_test.go`). Keep the substring re-filter and any ancestor/path helper pure and unit-tested; the live `Search`/`cmdFind` path has no coverage because no client mock/seam exists â€” same debt noted by the `id:` and JSON tickets; a client interface seam is a worthwhile future ticket.
