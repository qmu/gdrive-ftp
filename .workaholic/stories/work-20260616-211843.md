# Branch Story: work-20260616-211843

## 1. Overview

A small follow-up branch hardening `gdrive-ftp`'s file-transfer behavior: it
documents that transfers are byte-for-byte binary, and fixes a data-loss bug
where `put` to an existing folder created a same-named file shadowing the folder
instead of uploading into it.

## 2. Highlights

- **Fixed:** `put <file> "/drive/Folder"` (no trailing slash) now uploads **into**
  the folder under the local basename, instead of creating a same-named file in
  the parent that shadowed the folder and wedged the path.
- **Documented:** README now states transfers are byte-for-byte binary (no
  base64/conversion), with the short-download length check called out.

## 4. Outcome

`put`/`get` are now symmetric: both move a file *into* a target directory when
the destination is a folder. The previous behavior created a file named after
the folder in its parent — and because `Upload` replaces a single exact-name
match, repeated `put`s silently overwrote each other, while the resulting
folder+file name collision wedged every name-based command (`ls`/`rm`/`cd`) with
`ambiguous name`. The fix uses `resolveDir(dest)` as the authoritative
"is this an existing folder?" probe (one extra round-trip on the happy path,
chosen over a string heuristic). Verified live against Drive: two sequential
`put`s into a folder land inside it with no stray parent file; rename and
trailing-slash forms still work. The README binary-transfer note makes the
tool's key differentiator explicit. `go build`/`vet`/`test` green.

## 5. Historical Analysis

The bug was reported from real use (moving two coverage-matrix PDFs into
`/My Drive/00_作業`) immediately after the prior branch shipped — a good example
of dogfooding surfacing a foot-gun the feature work missed. The fix deliberately
mirrors `get`'s existing into-directory semantics (documented in the prior
branch's virtual-root work), keeping the two commands symmetric rather than
inventing new `put`-only behavior.

## 6. Concerns

### Wedged paths from the old bug still need out-of-band cleanup

- **Severity:** low
- **Description:** A path already broken by the old behavior (a folder and a
  same-named file coexisting) cannot be cleaned with the CLI, since `rm`/`get`
  match by name and refuse the ambiguous target (`internal/shell/commands.go`,
  `internal/gdrive/client.go` `FindOne`/`ErrAmbiguous`). The fix prevents new
  occurrences but does not heal existing ones.
- **How to Fix:** Add an ID-based selector (e.g. `rm --id <fileId>` / `get --id`)
  so an already-wedged name can be disambiguated and removed from the CLI.

### put-destination logic is verified live, not unit-tested

- **Severity:** low
- **Description:** The destination decision is network-bound (`resolveDir` →
  `FindDir`) and the repo has no Drive-client mock, so this regression was
  verified by live round-trip, not an offline test (`internal/shell/commands.go`
  `cmdPut`). Only pure helpers are unit-tested in this repo.
- **How to Fix:** Introduce a client-interface seam so `cmdPut`/`cmdGet`
  destination logic becomes table-testable offline with a fake provider.

## 7. Successful Development Patterns

- **Authoritative probe over string heuristic** — using `resolveDir(dest)` to
  detect a folder (rather than guessing from the path) means the fix also
  resolves the wedged folder+file collision toward the folder, because `FindDir`
  filters on `mimeType = folder`.
- **Live round-trip verification with cleanup** — reproduced the exact reported
  scenario against Drive (two puts into a folder), confirmed the fix, and trashed
  the test artifacts, mirroring the binary-fidelity verification approach.
