# Branch Story: work-20260616-073652

## 1. Overview

This branch grew an FTP-style Google Drive CLI from a My-Drive-only loopback
tool into a remote/SSH-friendly, Shared-Drives-aware client and packaged it for
public distribution. It replaced the loopback OAuth flow with a terminal
copy/paste consent flow that works identically over SSH, taught the client to
enumerate and navigate Shared Drives behind a synthetic virtual root, added
sftp-style and zsh Tab completion, and shipped the tool as a dual-agent
(Claude + Codex) plugin with an MIT license and a usage skill.

## 2. Highlights

- Replaced loopback OAuth with a terminal copy/paste consent flow (OSC 52
  clipboard, tmux-aware, single-keypress) that became the only path and works the
  same locally and over SSH.
- Added Shared Drives support to the gdrive client and a synthetic virtual root
  listing My Drive plus every accessible Shared Drive, with the drive name as the
  first path segment.
- Fixed empty Shared Drive listings by carrying `DriveID` on enumerated drives so
  corpus scoping resolves correctly.
- Added sftp-style in-shell Tab completion plus a zsh prompt-completion script for
  command verbs, remote Drive paths, and local paths.
- Packaged the repo as a public dual-agent Claude + Codex plugin with an MIT
  LICENSE, a `gdrive-ftp` usage skill, and install manifests.

## 3. Changes

**Why:** The CLI worked only on a local machine and only against My Drive — its
loopback OAuth flow broke over SSH (the laptop browser can't redirect to
`127.0.0.1` on a remote host), and every listing keyed by a folder ID under the
default corpus, leaving Shared Drives invisible. The goal was to make the tool
usable from any terminal (including remote/tmux sessions), expose the full drive
hierarchy, and then make it discoverable/operable by other agents and the public.

The work moved in three phases: **remote-friendly auth** → **Shared Drives** →
**ergonomics & packaging**.

## 4. Outcome

Transforms `gdrive-ftp` from a My-Drive-only, loopback-OAuth CLI into a
Shared-Drive-aware, remote-friendly tool publishable as a Claude Code + Codex
plugin. Remote OAuth now copies the consent URL to the local clipboard via OSC 52
and accepts a pasted `127.0.0.1` redirect URL (`codeFromRedirect` validates
`state`, surfaces `error=`, rejects empty `code`, bare-code fallback). The
`gdrive` wrapper gained `ListDrives` and a `driveID` parameter threaded through
listing/lookup/upload, keeping `drive/v3` types behind the domain `Ref`. The shell
starts at a virtual root over `My Drive` + Shared Drives, with `DriveID` carried on
`cwd[0]`. Tab completion (in-shell via `golang.org/x/term`, plus a zsh script)
covers verbs, live remote paths, and local paths. A pure-docs skill documents
one-shot usage, and the repo is packaged as a dual-agent plugin (all manifests
`1.0.0`, MIT). Stdlib-only posture preserved across the branch; new tests cover the
auth and completion/path helpers.

## 5. Historical Analysis

A young project — the meaningful history is *intra-branch*: each ticket built on
the prior one. Shared-Drive client support was deliberately split as the
foundation for the virtual root, which became the cwd/`currentDriveID`/`driveList`
substrate the Tab completion reused, which the skill documented, which the plugin
packaging distributed. The foundation passed `driveID == ""` everywhere so My
Drive stayed identical until the consumer ticket landed — keeping every commit
green. Later tickets mirrored earlier idioms (the OAuth `state`/`error`/empty-code
validation; the `openBrowser` best-effort pattern; `golang.org/x/term` reused for
the line editor; `../workaholic` manifests copied verbatim) and reaffirmed the
conservative-dependency posture and anti-corruption `Ref` boundary throughout.

## 6. Concerns

- **Skill/README drift risk (moderate)** — the publicly-distributed skill
  duplicates behavior derived from `README.md`/source; if verbs, export-format
  mappings, or path semantics change without updating `SKILL.md`, downstream
  agents act on stale instructions. *Fix:* keep README the single source of truth,
  add a reminder to update `SKILL.md` and bump all four manifests in lockstep on
  CLI changes.
- **Tab-completion latency / no caching (low)** — each remote Tab is a live `List`
  round-trip with no caching. *Fix:* optional per-session `(driveID, folderID)`
  cache, invalidated on `cd`/mutation.
- **Unterminated quotes silently disable Tab completion (low)** — an open quote
  makes `tokenize` error and Tab no-ops. *Fix:* surface a brief hint instead of
  doing nothing.
- **OSC 52 copy can be silently swallowed (low)** — terminals that don't honor
  OSC 52 fail the copy silently; the printed URL fallback is present. *Fix:* note
  copy is best-effort.
- **Public repo exposes ticket author emails (low)** — publishing makes
  `.workaholic/tickets/` (with author emails) public; left as-is to match the
  `../workaholic` convention. *Fix:* confirm intended, or exclude/scrub for the
  public surface.

## 7. Successful Development Patterns

- **Dependency-ordered ticket splitting that keeps every commit green** —
  foundation (`driveID == ""`) before consumer (virtual root) meant no broken
  intermediate state.
- **Pure-core extraction for testability** — `codeFromRedirect`, OSC 52 encoder
  (tested via an `os.Pipe`), the gdrive query helpers, and the completion helpers
  behind a fake provider — all unit-tested without network or a TTY.
- **Reusing in-repo precedent over new dependencies** — `golang.org/x/term`,
  reused validation phrasing, the `openBrowser` idiom, verbatim `../workaholic`
  manifests; the dependency surface stayed stdlib-only.
- **`crlfWriter` as a non-invasive raw-mode shim** — localized the raw-mode
  `\n`→`\r\n` concern to one adapter, leaving all command code untouched.
- **Recognizing when heavyweight tooling is unnecessary** — the pure-docs skill
  needs no `scripts/build-plugins`/`dist/` generator; one `skills/` dir serves
  Claude, Codex, and the skills CLI directly.
